# **Systems Engineering of Clawsmith: A Persistent, Multi-Room Agent Floor Architecture on OpenClaw**

## **1. Architectural Foundation of the Cozy, Multi-Room Agent Floor**

Transitioning from standard chatbot threads to a visual, persistent multi-agent dashboard requires a robust local runtime. The system is built directly on top of the open-source, local-first OpenClaw gateway. The local OpenClaw gateway binds to ws://127.0.0.1:18789 and acts as the secure central control plane. It manages messaging interfaces, isolates runtime sessions, and handles event streaming directly to your local hardware, protecting data privacy.

[Full diagram and description of OPENCLAW GATEWAY with Control Plane WS, Session Routing, Event Bus, Modular Cell Directory (~/.openclaw/workspace/rooms/<room_id>/), Total-ReClaw Engine (SQLite Vector + FTS5), WebSocket State Hub]

Rather than a complex, interconnected dynamic mesh, this architecture organizes your operating floor into static, isolated "cells" (project rooms). Each cell is structured as a dedicated folder in the OpenClaw workspace directory.

This approach keeps context completely isolated, preventing token bloat, data leakage, and system-wide crashes.

For long-term persistence across active runs, the operating floor relies on the Total-ReClaw hybrid memory engine. Total-ReClaw integrates directly with the gateway as a local memory plugin, using the native better-sqlite3 and sqlite-vec libraries. This setup bypasses the compilation bugs and model limitations of older memory systems, dynamically recording up to five key interaction points per turn (capping records at 2,000 characters to keep context clean).

To retrieve the most relevant information for active tasks, Total-ReClaw runs combined semantic and relational (FTS5) queries, scoring stored memories based on cosine similarity, age, and retrieval frequency.

[Full scoring formula with semantic cosine similarity, decay coefficient, elapsed time, importance rank, access boost coefficient]

A local background thread consolidates stored memories every 360 minutes. This process merges entries older than seven days that share over 85% similarity, keeping search performance sharp. Injected memories are explicitly marked with a trust="unverified" metadata tag, keeping agents informed of memory source boundaries and preventing potential prompt-injection attacks from altering system configurations.

## **2. The Clawsmith Meta-Agent: Modular Team Synthesis**

The **Clawsmith** is a local meta-agent that acts as a workflow compiler, translating plain-English instructions into structured agent cells.

[Plain-English Prompt] ---> [Clawsmith Compiler] --->
                                                      |-- > SOUL.md (Room Rules)
                                                      |-- > AGENTS.md (Worker Desks)
                                                      |-- > Dynamic Tool Configurations

When given a new goal, the Clawsmith determines how many agents are needed, outlines their specific roles, and writes the required configurations. To prevent unnecessary API costs and performance bottlenecks, Clawsmith uses a pruning step to merge overlapping roles, ensuring your agent teams remain small and efficient.

[Full TypeScript interfaces for AgentDesk and CellBlueprint, and the ClawsmithCompiler class with compileGoalToCell, pruneOverengineeredWorkers (example with scrapers + parsers consolidation), analyzeGoalRequirements, applyVisualTheme (Grant Hall Dungeon for grant/fund rooms, Hacker Den Terminal otherwise)]

This multi-room floor implements a strict supervised hierarchy:

* **Main Boss Agent**: Handles user commands, organizes summaries, and routes tasks to designated Room-Chiefs.
* **Room-Chiefs**: Act as localized coordinators. They receive goals from the Boss Agent, task local desk-anchored workers, compile results, and display summaries on visual room boards.
* **Specialist Workers**: Handled as desk-anchored sprites. These workers execute concrete steps like scraping, parsing, drafting, or packaging. They have access only to the specific tools required for their task, preventing out-of-bounds actions.

Communication paths are strictly parent-to-child: workers communicate only with their Room-Chief, eliminating unnecessary chatter and keeping operations organized and predictable.

## **3. Cell-Based Room Lifecycle Model**

The system uses a simple, folder-based cell lifecycle. Because workers remain anchored to their rooms, adding, updating, or deleting cells is fast, lightweight, and completely safe.

[Full lifecycle diagram: Clawsmith Compilation -> 1. CELL CREATION (Create subfolder, Generate config files, Init local memory) -> 2. DESK ANCHORING (Set sprite profiles, Map task callbacks, Open WebSocket pipe) -> 3. RUNTIME EXECUTION (Listen for changes, Trigger visual steps, Capture outputs) -> 4. CELL REMOVAL (Log final outputs, Purge room folder, Clear system memory)]

* **Creation**: The Clawsmith creates a dedicated subdirectory under ~/.openclaw/workspace/rooms/<room_id>/ and writes standard workspace Markdown configurations, including SOUL.md, AGENTS.md, and USER.md.
* **Anchoring**: The frontend dashboard reads the local AGENTS.md and renders the designated pixel-art sprites directly to their static workstations inside the room, initializing their WebSocket listener channels.
* **Execution**: Workers run assigned tasks, streaming events to the frontend and updating their visual states. High-risk actions are paused at approval gates, holding execution until the user clicks to approve.
* **Removal**: Deleting a room simply requires purging its directory. Active processes are safely shut down, and final summaries are recorded in the global long-term memory engine, keeping the dashboard clean.
